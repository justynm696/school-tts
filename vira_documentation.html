Section 01

📋 XML Data Structure

Structured representation of the V.I.R.A. system data schema in XML format

vira_schema.xml
<?xml version="1.0" encoding="UTF-8"?>
<!-- V.I.R.A. CeltechVoice Application - Data Schema -->
<database name="celtech_vira" version="1.0" charset="utf8mb4">

  <tables>

    <!-- EVENTS: School events and announcements -->
    <table name="events" engine="InnoDB">
      <field name="id" type="VARCHAR(10)" key="PRIMARY"/>
      <field name="title" type="VARCHAR(255)" null="false"/>
      <field name="content" type="TEXT" null="false"/>
      <field name="event_date" type="DATE" null="false"/>
      <field name="category" type="VARCHAR(50)"/>
      <field name="priority" type="ENUM" values="high,medium,low" default="medium"/>
      <field name="is_active" type="BOOLEAN" default="true"/>
      <field name="created_at" type="TIMESTAMP" default="CURRENT_TIMESTAMP"/>
    </table>

    <!-- USER_PREFERENCES: User settings stored client-side -->
    <table name="user_preferences">
      <field name="user_id" type="VARCHAR(50)" key="PRIMARY"/>
      <field name="theme" type="ENUM" values="light,dark" default="light"/>
      <field name="voice_id" type="VARCHAR(100)" key="FOREIGN" ref="voice_settings.voice_id"/>
      <field name="speech_rate" type="DECIMAL(2,1)" check="0.5 TO 2.0"/>
      <field name="search_history" type="JSON"/>
    </table>

    <!-- NAVIGATION_DATA: Interactive campus floor plan data -->
    <table name="navigation_data">
      <field name="id" type="INT AUTO_INCREMENT" key="PRIMARY"/>
      <field name="floor" type="INT" null="false"/>
      <field name="name" type="VARCHAR(255)"/>
      <field name="type" type="ENUM" values="office,lab,classroom,facility,amenity"/>
      <field name="coordinate_x" type="INT"/>
      <field name="coordinate_y" type="INT"/>
      <index type="FULLTEXT" cols="keywords,description"/>
    </table>

  </tables>

  <relationships>
    <relation from="user_preferences.voice_id" to="voice_settings.voice_id" type="ManyToOne"/>
    <relation from="search_logs.user_id" to="user_preferences.user_id" type="ManyToOne"/>
    <relation from="tts_usage_logs.voice_id" to="voice_settings.voice_id" type="ManyToOne"/>
  </relationships>

</database>

Key Relationships

📅
EVENTS → TTS_USAGE_LOGS 1:N
One event can be played multiple times; each log references a single event.
👤
USER_PREFERENCES → VOICE_SETTINGS N:1
Multiple users may share the same voice; each user has one active voice preference (FK: voice_id).
🔍
USER_PREFERENCES → SEARCH_LOGS 1:N
One user can perform many searches; each search log belongs to one user session.
🔊
VOICE_SETTINGS → TTS_USAGE_LOGS 1:N
One voice configuration can power many TTS sessions logged over time.

⬤ Level 2 — Detailed Process Breakdown

P1: Search Engine

  • 1.1 Input Validation (sanitize, lowercase)
  • 1.2 Search Algorithm (titles, content, nav)
  • 1.3 Result Ranking by relevance
  • 1.4 DOM Rendering of result cards

P2: Content Display

  • 2.1 Category Filter (tab selection)
  • 2.2 Data Retrieval from schoolData[]
  • 2.3 Card Generation (HTML elements)
  • 2.4 DOM Rendering with animations

P3: Voice Assistant

  • 3.1 Microphone Access & permission
  • 3.2 Speech Recognition API stream
  • 3.3 Command Parsing & intent detection
  • 3.4 Action Execution (search/navigate/TTS)

P4: TTS Engine

  • 4.1 Content Extraction (text/title)
  • 4.2 Settings Retrieval (voice, rate, vol)
  • 4.3 Utterance Creation (SpeechSynthesis)
  • 4.4 Speech Synthesis + Progress bar

P5: Interactive Maps

  • 5.1 Page Initialization (floor 1 default)
  • 5.2 Floor Data Retrieval by floor number
  • 5.3 Map Rendering + hotspot markers
  • 5.4 Location Search & highlight

Cross-Cutting Flows

  • Theme Toggle → LocalStorage → CSS class
  • Debounced Search (300ms pause)
  • PWA: Cache → Offline → Background sync
  • Error handling → Reset → Retry
